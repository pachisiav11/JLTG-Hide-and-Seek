# ┬¦R ŌĆö Redundant: still on the superseded path

### R1. The Transit layer is the thing ┬¦G0 exists to replace, and it is on by default **[V]**
`src/features.js:36`, wired at `src/app.js:195-196`.

`lines.js:4-8` and `overpass-lines.js:7-10` both state the finding plainly: Google's
`TransitLayer` is raster tiles from Google's own feed inventory, it takes no options, and in
Mumbai it draws the Metro but **not** the suburban locals ŌĆö the lines that actually decide
the game. That is the entire reason ┬¦G1 built the Overpass line pipeline.

The app then turns that superseded layer on at boot (`features.setTransit(true)`, with the
comment "hiders often stay near transit, so seekers want it visible from the start") and
leaves the real one (­¤Üä Rail) behind a button that must be pressed, waits on a slow fetch,
and fails ~64% of individual Overpass calls.

So the default view is the incomplete raster one, and the complete vector one is opt-in.
Worse, when both are on they draw the same corridors twice in different colours with no
indication that one of them is missing agencies.

**What to decide:** either default ­¤Üä Rail on (and accept the fetch at board-open), or keep
­¤Üå Transit as the instant-but-incomplete preview and say so in its tooltip. What is not
defensible is the current state, where the known-incomplete source is the one a player sees
without asking.

### R3. Two different, contradicting definitions of "1st / 2nd admin division" ship together **[V]**

**Definition A** ŌĆö `src/places.js:103-107`, `DDS_ADMIN_LEVELS`, used by the ­¤Ś║ Admin check
tool (`layers.js:795`) and by `extractAdminLevels` (`places.js:131`):

```
administrative_area_level_1  ŌåÆ  "1st Admin division"
administrative_area_level_2  ŌåÆ  "2nd Admin division"
locality                     ŌåÆ  "City / locality (Ōēł3rd)"
```

**Definition B** ŌĆö `overpass-lines.js:97+`, `COUNTRY_DIVISION_LEVELS`, used by Measuring Ō¢Ė
1st/2nd Admin Division Border via `divisionOrdinal` (`lines.js:462`):

> Singapore `[5, ŌĆ”]`, Ireland `[5, 6]`, Hong Kong `[5, 6]`, Japan `[4, 7]`, Portugal `[6]`,
> Germany `[4]` (no 2nd division at all), USA `[4]`, UK `[4]`, Philippines ŌĆö omitted
> entirely.

Definition B is the one the repo *measured*, over 1100 grid probes plus 271 city probes,
and the commit log is a record of it overturning the naive answer four separate ways
(maritime boundaries, territorial water, border straddle, city-states). Definition A is
Google's naming convention, assumed to mean the same thing.

They disagree on real boards. In Japan, Admin check's "County / 2nd admin" row reads
Google's `administrative_area_level_2`, while the Measuring border card correctly draws
level 7 (municipality). In Singapore, Admin check reports a level-1 that the measurement
says is level 5. In Germany the border card says *there is no 2nd division* and falls back
to hand-drawing, while Admin check confidently prints a Ō£ō or Ō£Ś for one.

A seeker who uses ­¤Ś║ Admin check to reason about a division question and then asks it is
being told two different things by the same app. The tool is a "diagnostic, not an
elimination" (`layers.js:790`), which limits the blast radius ŌĆö but a wrong diagnostic that
contradicts the app's own elimination is worse than no diagnostic.

**Fix direction:** Admin check should resolve its 1st/2nd rows through
`loadCountryDivisions()` / `COUNTRY_DIVISION_LEVELS` like the border cards do, and label
rows it cannot resolve as "ŌĆō" rather than borrowing Google's numbering.

### R4. Matching Ō¢Ė admin1ŌĆōadmin4 hand-trace, helped by a DDS layer that only reaches 3 **[V]**
`src/layers.js:1180` `_matchRegion`, `layers.js:1213` `_adminTracePrompt`, `places.js:103`.

The four admin-division Matching cards are `mode: "region"` ŌĆö the seeker outlines their own
division by hand. The tracing helper offers to highlight the official boundary via DDS, but
`DDS_ADMIN_LEVELS` has exactly three entries, and the code says why:

> "Google's Data-Driven Styling only exposes FeatureLayers for these (there is no level-3/-4
> FeatureType), so the tracing helper can only draw these boundaries even though the game
> asks about 1stŌĆō4th divisions."

cannot set, so in practice it degrades to a plain draw prompt.

Meanwhile `lineGeometry(kind, gameArea, { divisionOrdinal })` (`lines.js:462`) already
returns real boundary geometry for ordinals 1 and 2 from OSM, worldwide, with no Map ID and
no DDS.

**Fix direction:** route admin1/admin2 Matching through `divisionOrdinal` the way the
Measuring border cards do ŌĆö the region can be derived from the boundary rather than traced.
admin3/admin4 have no measured level table and should stay hand-drawn, but their trace
helper should say "no official outline available for this level" instead of offering the
wrong three.

### R5. The Socket.IO multiplayer relay is live in production and has no client **[V]**
`server.js:180-250`, plus the `socket.io` runtime dependency and the `SocketIOServer`
import at `server.js:19`.

`REVIEW_FINDINGS.md` closes ┬¦E4 as "multiplayer, out of scope" and `SESSION_LOG.md` ┬¦1.2

process the Overpass proxy needs to stay responsive on.

**Fix direction:** delete the relay block and the `socket.io` dependency, or gate it behind
`process.env.MULTIPLAYER_ENABLED`. Reviving it later is a git revert.

### B1. A Metro Lines question always reads `closest "?"` in the Questions panel **[M]**

> **Measured 2026-07-18.** `describeStep` on a Metro Lines step built exactly as
> `layers.js:1413` builds it returns:
> ```
> Tentacles ┬Ę Metro Lines ┬Ę closest ŌĆ£?ŌĆØ (25 km)
> ```


`layers.js:1413` stores a line-mode tentacles step as:

```js
const inputs = { category: cat.id, categoryLabel: cat.label, radius: cat.radius,
                 lines: stored, center };
```

`tools.js:785-791` describes every tentacles step as:

```js
const sel = (step.inputs.features || [])[step.answer?.featureIndex];
return `Tentacles ┬Ę ${cat} ┬Ę closest ŌĆ£${sel?.name || "?"}ŌĆØ (${rTxt})`;
```

`inputs.features` is `undefined` for a line step, so `sel` is `undefined` and every Metro
Lines question in the panel reads **`Tentacles ┬Ę Metro Lines ┬Ę closest "?" (25 km)`** ŌĆö
identical for every line the seeker could have picked.

`tools.js:503` already knows the shape (`const isLines = Array.isArray(lines) && lines.length > 0`);
`describeStep` was simply never taught it. The `matching` branch immediately above
*was* ŌĆö `tools.js:778-781` handles `nearestLine` by looking up `inputs.lines`.

**Impact:** the panel is how a seeker decides which question to disable when the board looks
wrong. With several Metro Lines questions asked, they are indistinguishable, and the rename
default (`layers.js:782` seeds `promptText` with `describeStep(step)`) carries the `"?"`
into the name if they rename it.

**Fix:** mirror the matching branch ŌĆö

```js
if (step.answer?.none) return `Tentacles ┬Ę ${cat} ┬Ę none within ${rTxt}`;
const items = step.inputs.lines || step.inputs.features || [];
const sel = items[step.answer?.featureIndex];
return `Tentacles ┬Ę ${cat} ┬Ę closest ŌĆ£${sel?.label || sel?.name || "?"}ŌĆØ (${rTxt})`;
```

(`lines` entries carry `label`, `features` entries carry `name` ŌĆö hence both.)

### B2. A nearest-line question with ONE line is accepted, and does nothing **[M]**

> **Measured 2026-07-18** on a 195.6 km┬▓ board with one drawn line through it:
> ```
> match=true  -> eliminated null      of 195.6 km2   (silent no-op)
> match=false -> eliminated 195.6 km2 of 195.6 km2   (whole board)
> ```
> Neither throws, so neither is marked failed or banner'd. The "differ" case is the more
> alarming of the two: the seeker answered correctly and the board goes fully eliminated,
> which surfaces as "check the most recent answer" ŌĆö advice pointing away from the cause.


`layers.js:1128`: `if (lines.length < 1) { ŌĆ” return this.openPanel(); }` ŌĆö so **one** drawn
line proceeds to the selection sheet and is committed.

Follow it through `tools.js:446` `matchingNearestLine` ŌåÆ `lineCells(lines, gameArea)`:
with a single line, every densified seed has `owner = [0]`, so `perOwner[0]` collects every
Voronoi cell, `turf.union` merges them into the whole padded bbox, and
`cells[0] = safeIntersect(merged, gameArea)` is **the entire game area**.

Then:
- answer *match* ŌåÆ `safeDiff(gameArea, gameArea)` ŌåÆ `null` ŌåÆ eliminates nothing. The
  question is committed, enabled, listed in the panel, and contributes zero shading. It does
  not throw, so it is never added to `failedSteps` and never appears in the banner.
- answer *differ* ŌåÆ `eliminated = cells[0]` = the whole board ŌåÆ the board goes fully
  eliminated, `computeActiveArea` returns `EMPTY_AREA`, and the seeker gets "Every question
  combined rules out the whole area ŌĆö check the most recent answer" for a question that was
  answered correctly.

Both outcomes are wrong and neither names the cause. The Tentacles line picker already got
this right ŌĆö `layers.js:1395` warns when `lines.length < 2` that the question "can't
distinguish between lines" ŌĆö because there the radius still makes a 1-line question
meaningful (it degenerates to a radar). In *Matching* there is no radius, so a single line
is not degenerate-but-useful, it is meaningless: "is your nearest line the same as mine"
when there is only one line has exactly one possible truthful answer.

**Fix:** require `minPts`-style two lines in `_matchNearestLine` ŌĆö change the guard to
`lines.length < 2` with a toast saying why, matching how `_matchNearest` enforces
`minCount: 2` (`layers.js:1008`).

### B3. The candidate radio list pre-checks index 0, so the "require an explicit pick" guard rarely binds **[V]**

`layers.js:513` builds every feature radio list with:

```js
<input type="radio" name="${name}" value="${i}" ${i === 0 ? "checked" : ""}/>
```

Two `Add` handlers then guard against a missing selection, each with a comment describing
the bug it is there to prevent:

- `layers.js:1042` ŌĆö "Require an explicit pick. Defaulting to `0` here was the second mouth
  of the same bug: with the checked item filtered away, Add silently recorded a station the
  seeker never chose and could not see."
- `layers.js:1463` ŌĆö same reasoning for Tentacles.

But `find(r => r.checked)` only returns `undefined` when *something cleared the default* ŌĆö
which is `repairRadioSelection` (`ui.js`), and that only runs when the user typed in the
filter box (`_wireFeatureSearch`, `layers.js:531`). On the common path ŌĆö open the sheet,
scan a list of six stations, press Add without touching the filter ŌĆö **candidate #1 is
already checked and is silently committed.**

That is the original failure with a narrower mouth, not a closed one: the seeker never made
a choice, and `describeStep` will report a station name they did not pick.

The fix the code already believes it made is one character: drop `${i === 0 ? "checked" : ""}`
from `_featureListHTML`. The guards downstream are then load-bearing on every path, and
`repairRadioSelection`'s "otherwise clear it and require an explicit pick" branch becomes
the whole rule rather than a special case.

Note `_chooseTentacleLines` (`layers.js:1386`) already builds its list with **no** default
checked ŌĆö so the two pickers behave differently today (see B9).

### B4. A failed zone union silently deletes the play area **[V]**
`src/zones.js:129` and `:143`:

```js
store.update((g) => {
  g.zones.push(zone);
  g.gameArea = unionRings(g.zones.map((z) => z.polygon));
});
```

`unionRings` (`geo.js:135-146`) catches its own exception and returns `null`. So a turf
union failure on the *newly added* zone assigns `gameArea = null` ŌĆö throwing away a board
that was valid a moment ago, with every zone still listed in the panel.

Downstream, `layers._render` (`layers.js:271`) does handle a null `gameArea` gracefully and
raises "N questions are saved but can't be shown without a play area ŌĆö add a zone (Zones Ō¢Ė
Draw) to see them again". That message is *wrong here*: the seeker has zones, they just
added one, and the advice is to do the thing that broke it.

The same shape in `removeZone` (`:143`) is less severe (removing a zone genuinely can empty
the board) but is indistinguishable from the failure case.

**Fix:** compute into a local, and only commit when it succeeds:

```js
const merged = unionRings(g.zones.map((z) => z.polygon));
if (!merged && g.zones.length) { /* keep the old gameArea, toast the failure */ }
else g.gameArea = merged;
```

D1/D2 already refuse the two *known* causes of a bad ring at draw time, which is why this
has not been seen ŌĆö but `unionRings` returning null is exactly the "swallowed, then
assigned" pattern that `computeActiveArea` was fixed for in the last changelist
(`tools.js:744-754`), and it is still here in the zone layer.

### B5. Measure mode competes with every pick and draw flow for the map click **[V]**

`features.js:28` registers a permanent map click listener in `init()`:
```js
this.map.addListener("click", (e) => this._onClick(e.latLng));
```
`_onClick` returns early unless `measure.active`, so it is inert *while measure is off*.

But nothing turns measure off when a question flow starts. `layers.pick()`
(`layers.js:395`), `layers._drawShape()` (`:447`) and `zones.startDraw()` (`zones.js:44`)
each add their **own** map click listener. With ­¤ōÅ Measure toggled on, tapping the map to
place a radar centre also drops a numbered measure marker and, on the second tap, draws a
yellow measurement line and shows the readout ŌĆö over the question being set up.

Worse in the other direction: `_onClick` calls `this._clearMeasure()` when two points already
exist, so a long drawing session leaves markers appearing and vanishing under the drawn
polygon.

`features.clearAll()` (`:95`) does reset measure ŌĆö but it is only called on a *game switch*
(`app.js:185`) and on Clear board (`games.js:~128`), never when a tool flow begins.

**Fix:** have `pick`, `_drawShape` and `startDraw` suspend measure mode for their duration
(the same way they set `draggableCursor: "crosshair"` and restore it in `cleanup`), or have
`_onClick` bail whenever a `.draw-bar` is present in the DOM ŌĆö that element is the existing,
already-unique signal that a map-tap flow owns the click.

### B6. `clearBoard` leaves the redo stack and the rail filter behind **[V]**


### B7. Rail lines are drawn once and never re-rendered within a game **[V]**

`Lines.render()` (`lines.js:553`) is called from exactly three places: `openPanel`
(`:593`), `_setFilter` (`:582`), and the mode/line checkbox handlers. Unlike `Zones`,
`Layers` and `Focus`, the `Lines` class **never subscribes to the store** ŌĆö compare
`zones.js:35`, `layers.js:126`, `focus.js` (`store.subscribe(() => this.render())`).

Consequences within a single game:
- The payload is fetched once for the board bbox at the moment ­¤Üä was first pressed
  (`lines.js:526`: `if (this.data) return this.data;`). Add or remove a zone afterwards and
  the board changes, but the rail geometry does not ŌĆö it still shows the old bbox's lines,

- Palette changes (`jltg:palette`) restyle zones, layers and guides live but not the rail

The game-switch case *is* handled (`app.js:186-189` calls `lines.clear()`), which is why
this reads as an oversight in the within-game case rather than a missing concept.
